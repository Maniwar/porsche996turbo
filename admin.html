Data retention

What happens to old conversations, actions, and funnel events

The default is keep everything forever — no automatic deletion is scheduled. Pruning is a deliberate act you run from here: it deletes conversations, register actions, funnel events, and email logs older than the horizon you choose. Orders and their attribution stamps always survive, so revenue reporting is unaffected — only the drill-down into pruned transcripts is lost. Why prune at all: transcripts hold customer PII, and a bounded horizon is the usual privacy/cost hygiene at scale.

What the concierge can do

Tools

Everything the concierge can do for a signed-in patron lives here. There are two kinds, and they're added in two different ways:

• Model tools run code on our side — read the register, cancel, re-send a confirmation, log a mending request. The bot decides when to call them. You control which are switched on and how they're instructed; creating a brand-new model tool needs a developer (it has to run real server logic — see TOOLS.md).
• Form tools hand the patron a little form in the chat and write the result to an order. These are pure configuration — you can create, edit, and remove them right here, no deploy.

Model tools

code-backed actions the bot can call

Turn one off to withhold it from the bot entirely, or rewrite its instruction to change when and how the bot uses it. A blank instruction uses the built-in default. Every tool is ownership-scoped and logged; turning a core tool off will noticeably dull the concierge. To add a new one, see the developer note in TOOLS.md — a model tool needs a handler in the concierge function.

Form tools

capabilities you can create — the bot hands these out in chat

This is how you give the concierge a new capability without code. A form tool renders inside the chat when the concierge emits its {{form:slug:serial}} token, so it's order-scoped (it carries a serial). Each binds to one write path (a register action such as update_shipping_address or update_gift_details) — pick it when you create the tool. Fields are the labelled inputs the patron fills in (JSON: name, label, type text · state · zip, required, maxlength); at least one is required. Submissions run through the same verified, audited path as the bot's own tools, and — because the patron types each labelled field — this is the safe way to collect things the model shouldn't free-type (like an address). Turn one off to stop the bot offering it. Full guide: FORMS.md.

Does the concierge still behave?

Evals

Scripted conversations replayed against the live concierge, scoring the behaviors that matter — it shows the commission button on a buying signal, never leaks [HOLD], doesn't loop in discovery, reads the register instead of guessing. Each scenario runs a few times and reports a pass rate, so a prompt or model change can't quietly bring back a bug you already fixed.

Attribution QA

End-to-end pipeline check — no model calls, no production impact

Round-trips the attribution pipeline with qa- test keys: writes a funnel beacon, a conversation, and an attributed order (serial-less, cancelled — it can never count as revenue), verifies the tier stamp and the order→conversation join, then deletes everything it created. qa- and eval- traffic is excluded from every Conversion metric, so QA never moves production numbers. Spec: ATTRIBUTION.md.

Run the deck

live model calls — costs a little each run

Two kinds of check. Deterministic ones are exact (the reply contains a token, asks at most one question, a tool actually ran) — cheap and stable. The fuzzy ones ("did it advance the sale?") go to a binary LLM judge (yes/no, pinned) on the server, so no API key ever touches the browser. Signed-in scenarios use your admin session, so they read a real register.

Scenarios

the behavior deck — editable, no deploy

A scenario is a short scripted conversation plus checks on the bot's reply for a turn. Prefer a deterministic check whenever the behavior is mechanically observable; reserve the judge for genuine judgment, and phrase its criterion as one concrete yes/no. Turn a scenario off to skip it in a run.
